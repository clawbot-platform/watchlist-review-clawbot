Here is a concrete phased architecture delta for:

* `watchlist-review-clawbot`
* `claw-identity`
* `clawmem`

It is designed around your current stable baseline, which is still promotable at **12/12** with **0 deterministic regressions**, **0 retrieval-required failures**, and **note_quality_pass_rate = 1**. 

---

# 1. Target architecture split

## `watchlist-review-clawbot`

Owns:

* request intake and review orchestration
* deterministic scoring policy
* decision thresholds and label selection
* explanation assembly for deterministic evidence
* batch/regression eval and promotion gates
* analyst-note generation as a downstream consumer of deterministic evidence

This repo should remain the **policy brain** for the watchlist workflow.

## `claw-identity`

Owns:

* canonical entity comparison services
* OFAC-linked identity evidence
* matched profile linkage and normalization
* relationship evidence derivation
* linked-document / identifier evidence
* same-pair and same-profile history lookups once stabilized

This service should become the **identity evidence engine**.

## `clawmem`

Owns later:

* retrieval and memory layer
* prior-case retrieval across large history
* example/snippet retrieval for note generation
* optional graph-context retrieval / semantic recall
* optional long-horizon institutional memory beyond transactional scoring

This should become the **memory/retrieval substrate**, not the first home of deterministic policy.

---

# 2. Guiding principle

Do **not** move everything out immediately.

The safest order is:

* build and validate deterministic features first in `watchlist-review-clawbot`
* extract reusable identity evidence generation into `claw-identity`
* integrate `clawmem` only after you know exactly what memory/retrieval patterns are worth operationalizing

That keeps scoring changes easy to test against the already-green suite.

## Phase 1 — Disposition Memory Scoring v1

This is your **Priority 1** and fastest win.

### Objective

Add deterministic scoring based on **prior reviewed dispositions**, but only when:

* the same screened entity and matched list UID are confidently linked to prior reviewed cases

### New score block

Add to `internal/scoring/types.go`:

* `PreviousDispositionScore int`
* `PreviousDispositionReasons []string`

And fold into total decision evidence.

### What stays in `watchlist-review-clawbot`

Build it here first.

### New / changed packages in `watchlist-review-clawbot`

#### `internal/history` or `internal/disposition`

New package.
Responsibilities:

* normalized alert history store access
* same-pair lookup
* same-profile lookup
* recency weighting helpers
* contradiction-pattern lookup later

Suggested files:

* `store.go`
* `types.go`
* `normalize.go`
* `lookup.go`
* `recency.go`

#### `internal/scoring`

Add:

* `previous_disposition.go`
* new score components
* policy hooks for same-pair history

Responsibilities:

* compute:

  * `previous_disposition_score`
  * `previous_disposition_reason[]`
* only apply score when identity confidence for the pair is above threshold

#### `internal/runtime`

Add:

* history fetch before final scoring
* include history evidence in review context

#### `internal/api`

Add review-context fields:

* `previous_disposition_summary`
* `history_evidence`

#### `internal/eval`

Add repeated-pair test rules and fixtures

### Data model

Start in Postgres, not Neo4j.

Suggested tables:

* `review_case_history`
* `review_pair_history`
* `review_disposition_events`

Minimum normalized columns:

* tenant_id
* case_id
* alert_id
* screened_entity_fingerprint
* matched_list_uid
* matched_program
* decision_label
* contradiction_pattern
* created_at

### Decision logic

Examples:

* same pair, prior confirmed/escalated, recent → positive boost
* same pair, prior false positive with same contradiction pattern → negative boost
* unresolved prior investigation → bias toward `investigate_next_step`

### Deliverable A — Disposition Memory Scoring v1

### Acceptance

## Phase 3 — Contradiction-Pattern Memory

This is your **Priority 3**.

### Objective

Strengthen deterministic handling of:

* sparse false positives
* recurring wrong-context alerts
* repeated contradiction signatures

### Best home initially

`watchlist-review-clawbot`

### New / changed packages

#### `internal/history`

Extend with:

* contradiction-pattern normalization
* pattern-frequency lookup
* pattern-outcome lookup

Suggested files:

* `patterns.go`
* `pattern_lookup.go`

#### `internal/scoring`

Add:

* `pattern_memory.go`

New fields:

* `ContradictionPatternScore`
* `ContradictionPatternReasons []string`

### Pattern examples

Normalized signatures:

* `name_only_overlap+country_mismatch+sparse_data`
* `dob_conflict+identifier_conflict+geography_conflict`
* `alias_match+exact_identifier_match`

### Decision behavior

* repeated same false-positive contradiction pattern → lower escalation likelihood
* repeated unresolved contradiction pattern → bias to `investigate_next_step`
* repeated true-positive corroboration pattern → positive support

### What later moves to `clawmem`

Only if you want long-horizon retrieval across much larger history:

* semantic similarity of contradiction patterns
* pattern retrieval across business lines
* example-based recall

For now, keep it deterministic and relational.

---

## Phase 4 — Extract stable evidence services into `claw-identity`

Do this only after Deliverables A and B are proven.

### Move to `claw-identity`

Reusable, policy-neutral evidence builders:

* normalized entity compare
* OFAC relationship evidence
* linked doc support
* same-pair prior history evidence
* same-profile prior history evidence

### Keep in `watchlist-review-clawbot`

Policy-owned logic:

* score weights
* thresholding
* label policy
* decision reasons
* eval and promotion gates

This keeps `claw-identity` reusable for future AML/sanctions clawbots.

---

## Phase 5 — Add `clawmem` deliberately

Not before.

### What `clawmem` should do later

* prior-case retrieval beyond direct pair history
* retrieval of relevant prior analyst decisions
* example retrieval for explanation generation
* optional graph-context retrieval
* optional Qdrant/Neo4j-backed neighborhood retrieval

### What `clawmem` should not do first

* it should not become the first scoring dependency
* it should not own deterministic policy
* it should not replace clean relational history for same-pair scoring

# 6. Recommended order of execution

1. **Disposition Memory Scoring v1**
2. **Contradiction-Pattern Memory v1**
3. **OFAC Relationship Scoring v1**
4. extract stable evidence generation into **`claw-identity`**
5. add broader retrieval support in **`clawmem`** only where it clearly helps

This gives you the fastest deterministic gains with the least architectural churn.

---

# 7. Bottom-line architecture decision

## Build first in `watchlist-review-clawbot`

Because:

* fastest iteration
* easiest to test
* directly protected by your existing promotion suite 

## Extract next into `claw-identity`

Because:

* identity/relationship evidence is reusable
* policy weights should stay local to the watchlist product

## Bring in `clawmem` later

Because:

* memory/retrieval is useful, but not required for first deterministic wins
* you do not need Neo4j/RAG to improve same-pair history or direct OFAC relationship scoring
